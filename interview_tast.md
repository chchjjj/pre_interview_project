# [사전 질문지] 지원자 최현지

## 1. Part A. 서버설계 작업 계획
### 요구사항
고객사(조직)별로 서로 다른 메시지 서비스 제공 업체를 사용해야 함. (클라우드메시지 or 센드톡)
### 가정 상황
서버(백엔드) - 화면설정(프론트엔드) 파트 담당자는 회의를 통해 API 문서를 확정함. (클라우드메시지, 센드톡)

### 작업 단계별 기술적 결정 사항과 리스크 대응 방안
[1단계] 수정 또는 추가할 DB 설계 회의
- 목표 : 효율적인 수정 또는 추가로 서비스 흐름에 문제 없는 DB 설계 (조직-업체 매핑)
- 산출물 : '발송 업체 정보(message_provider)' 테이블 별도 생성<br>

[기술적 결정 사항]
- 고민 : 조직별 발송 업체 정보를 어떻게 저장할 것인가?
- 대안 비교 : 기존 조직 테이블에 컬럼 추가 VS 발송 업체 정보 담은 테이블 별도 생성
- 선택 : 테이블 별도 생성
- 사유 : 차후 또다른 발송 업체가 추가될 경우를 대비하여 별도로 관리하는 것이 장기적으로 적합하다고 생각함.
---
[2단계] 조직별 어떤 업체를 선택했는지의 DB를 기반으로 한 발송 로직 고려
- 목표 : 메시지 발송 시 해당 사용자가 속한 조직의 업체 설정 값을 DB에서 조회
- 산출물 : 조직 관리자가 설정해둔 DB 내용에 맞춰 발송 로직이 세팅됨<br>

[기술적 결정 사항]
- 고민 : 업체마다 다른 API 규격을 어떻게 처리할 것인가?
- 대안 비교 : 발송 시점에 IF-ELSE 분기 처리 VS 공통 인터페이스/로직 세팅
- 선택 : 공통 인터페이스/로직 세팅
- 사유 : 
1) 업체별로 필드를 다르게 쓰는데, 코드에 직접 다 입력 시 유지보수가 힘들 것으로 예상하여 공통(표준) 규격 만드는 것이 합리적이라 생각함.
2) 발송 시점마다 개별 사용자가 업체를 선택하는 것이 아니라, 조직 관리자 판단 하에 조직 전체가 최소 며칠 ~ 몇 주 이상 한 업체만 사용할 것으로 판단함. 개별 사용자는 업체가 바뀌었는지 신경쓰지 않아도 되며 동일한 UI에서 발송하면 됨.
---
[3단계] 업체별 API 규격 통합 (구현)
* 목표 : DB에서 해당 조직이 사용하는 업체를 불러온 후 그 업체 규격에 맞게 데이터 (변환 및) 호출
---
[4단계] 예상되는 문제 상황과 대응 방안 마련
* 목표 : UX가 나빠지지 않도록 사전 방지하여 고객 만족도 유지
* 위험 요소 : 설정 변경 시점의 발송 정합성
* 영향 범위 : 관리자가 업체를 바꾸는 찰나에 발송 요청이 들어올 경우 처리 방식 혼선
* 완화 전략 : 
  1) 업체 변경 시 변경이 완료된(DB에 최종 반영된) 시점 이후의 요청부터 반영되도록 트랜잭션 처리 및 변경 이력 로그로 남기기
  2) 조직 관리자에게 사전 안내 또는 화면에 안내 문구를 띄워 (예. '변경 과정 중 발송된 메세지는 최종 변경 반영 전까지는 기존 업체로 발송됩니다.') 해당 내용을 인지하도록 하기. (조직 관리자는 재량에 따라 내부적으로 사용자들에게 발송 금지 시간을 사전에 안내할 수도 있음)<br><br>
